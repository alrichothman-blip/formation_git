Section 01

Qu'est-ce que le contrôle de versions ?

Imaginez que vous travaillez sur un document Word et que vous faites une modification désastreuse — vous souhaiteriez pouvoir revenir en arrière. La plupart des éditeurs de texte permettent un simple Ctrl+Z, mais que se passe-t-il si vous fermez le fichier ? Que se passe-t-il si vous êtes à dix étapes du problème ?

Un système de contrôle de versions (ou VCS, Version Control System) répond exactement à ce besoin : il enregistre toutes les modifications apportées à un fichier ou à un ensemble de fichiers au fil du temps, de sorte que vous puissiez rappeler des versions spécifiques plus tard.

💡
Bon à savoir

Le contrôle de versions ne concerne pas uniquement le code source. Il peut être utilisé avec n'importe quel type de fichier — graphiques, documents, données de configuration, etc.

Pensez à un VCS comme à une machine à remonter le temps pour votre projet : il vous permet de naviguer à travers toutes les versions de vos fichiers, de comprendre qui a modifié quoi et quand, et de fusionner le travail de plusieurs personnes en toute sécurité.

Section 02

Le problème sans VCS

Avant l'adoption des systèmes de contrôle de versions modernes, les développeurs avaient recours à des pratiques très artisanales — et potentiellement dangereuses.

La technique des copies manuelles

La solution la plus répandue consistait à dupliquer manuellement les dossiers :

Structure de fichiers typique (avant Git) 
# Exemple de nommage "artisanal" courant

projet_final/
projet_final_v2/
projet_final_VRAI_final/
projet_final_VRAI_final_2/
projet_final_definitif/
projet_final_definitif_UTILISER_CELUI_CI/

Cette approche pose de nombreux problèmes majeurs :

  • ✕ Confusion des versions : Impossible de savoir quelle version est la plus récente ou la plus stable.
  • ✕ Perte de contexte : Aucun moyen de savoir ce qui a changé entre deux versions.
  • ✕ Collaboration difficile : Deux développeurs qui travaillent sur le même fichier écrasent mutuellement leurs modifications.
  • ✕ Espace disque gaspillé : Chaque copie duplique l'intégralité du projet.
  • ✕ Aucune traçabilité : Impossible de savoir qui a modifié quoi, et pour quelle raison.
🔥
Scénario catastrophe

Vous travaillez à trois sur un projet. Développeur A modifie le fichier A, Développeur B modifie le même fichier. B envoie sa version par email. A reçoit l'email et écrase ses propres modifications avec la version de B. Résultat : des heures de travail perdues.

Section 03

Voici les modification que j'ai apporter

Contrôle de versions (Version Control)

Système qui enregistre les modifications apportées à un fichier ou à un ensemble de fichiers au fil du temps, permettant de rappeler des versions spécifiques ultérieurement. Il permet aussi de comparer les modifications, identifier les auteurs, et restaurer des états antérieurs.

Les concepts fondamentaux

📸

Commit (Instantané)

Une photographie de l'état de votre projet à un instant précis. Chaque commit capture les changements et inclut un message décrivant ce qui a été fait.

🌿

Branche (Branch)

Une ligne de développement indépendante qui permet de travailler sur une fonctionnalité sans affecter le code principal (souvent appelé main ou master).

🔀

Fusion (Merge)

L'opération qui intègre les modifications d'une branche dans une autre. Git est très intelligent pour résoudre automatiquement la plupart des conflits.

📦

Dépôt (Repository)

Le conteneur de votre projet. Un dépôt contient tous les fichiers du projet ainsi que l'historique complet de toutes les modifications.

🔄

Clone

Une copie complète et indépendante d'un dépôt, incluant tout l'historique. Avec Git, vous pouvez travailler entièrement hors ligne.

☁️

Remote (Distant)

Un dépôt hébergé sur un serveur (comme GitHub) qui permet à plusieurs développeurs de partager et synchroniser leur travail.

⚙️
Section 04

Comment Git fonctionne-t-il ?

Git est le système de contrôle de versions le plus utilisé au monde. Il a été créé en 2005 par Linus Torvalds — le créateur du noyau Linux — avec un objectif précis : rapidité, design simple, support fort pour le développement non-linéaire (milliers de branches parallèles), entièrement distribué.

La notion d'instantanés (snapshots)

Contrairement à d'autres systèmes VCS qui stockent les modifications sous forme de différences par rapport à la version précédente, Git pense ses données comme une série d'instantanés de votre système de fichiers miniature.

À chaque commit, Git prend une photo de tous vos fichiers et stocke une référence vers cet instantané. Pour être efficace, si des fichiers n'ont pas changé, Git ne les stocke pas à nouveau — il crée simplement un lien vers le fichier identique précédemment stocké.

A1B2
Commit 1
Init projet
C3D4
Commit 2
Ajout login
E5F6
Commit 3
Fix bug #12
G7H8
Commit 4
Release v1.0

Les trois zones de travail

📁 Le répertoire de travail (Working Directory)

C'est là où vous modifiez activement vos fichiers. Vos modifications ici ne sont pas encore enregistrées dans l'historique Git.

🗂 La zone d'index (Staging Area)

Un espace intermédiaire où vous préparez votre prochain commit. Vous choisissez précisément quelles modifications seront incluses dans le prochain instantané.

🗄 Le dépôt Git (Git Directory / Repository)

L'endroit où Git stocke les métadonnées et la base de données des objets de votre projet. C'est la partie la plus importante — c'est ce qui est copié lors d'un clone.

☁️ Le dépôt distant (Remote Repository)

Une version hébergée sur un serveur (GitHub, GitLab, Bitbucket…) qui permet la collaboration et la sauvegarde externe de votre historique.

Les commandes essentielles

Bash — Git workflow de base 
# 1. Initialiser un nouveau dépôt
git init

# 2. Vérifier l'état du répertoire de travail
git status

# 3. Ajouter des fichiers à la zone d'index (staging)
git add nom_du_fichier.js
git add .          # Ajoute tous les fichiers modifiés

# 4. Enregistrer un instantané (commit)
git commit -m "feat: ajout de la page de connexion"

# 5. Consulter l'historique des commits
git log --oneline

# 6. Créer une nouvelle branche et basculer dessus
git checkout -b feature/nouvelle-fonctionnalite

Section 05

Centralisé vs Distribué

Il existe deux grandes familles de systèmes de contrôle de versions. Comprendre leur différence est essentiel pour saisir pourquoi Git a révolutionné le développement.

Les systèmes centralisés (CVCS)

Dans un système centralisé (comme Subversion / SVN ou CVS), il existe un seul serveur central qui contient toutes les versions des fichiers. Les clients récupèrent des fichiers depuis ce serveur central.

⚡
Le point de défaillance unique

Si le serveur central tombe en panne, personne ne peut collaborer ou sauvegarder les modifications. Si le disque dur du serveur est corrompu sans sauvegarde récente, vous perdez tout.

Les systèmes distribués (DVCS)

Avec Git (Distributed VCS), les clients ne récupèrent pas seulement les dernières versions des fichiers — ils dupliquent intégralement le dépôt, historique compris. Chaque développeur possède une copie complète du projet.

Caractéristique Centralisé (SVN) Distribué (Git)
Travail hors-ligne Limité Complet ✓
Vitesse Dépend du réseau Opérations locales ultra-rapides
Redondance Point unique de défaillance Chaque clone = sauvegarde complète
Branches Coûteuses, lentes Légères, instantanées
Collaboration Flux linéaire, verrou de fichiers Non-linéaire, branches parallèles
Historique complet Uniquement sur le serveur Sur chaque machine locale
✅
Pourquoi Git a gagné

La combinaison vitesse + distribution complète + gestion des branches légères a rendu Git incontournable. Aujourd'hui, plus de 90% des développeurs professionnels utilisent Git comme système de contrôle de versions.

Section 06

Avantages concrets de Git

Git ne se contente pas de stocker vos fichiers — il transforme complètement la façon dont vous développez, collaborez et livrez du code.

Git conserve un historique complet de toutes vos modifications. Si un bug est introduit, vous pouvez identifier précisément quel commit l'a causé et revenir à un état antérieur stable en quelques secondes.

La commande git checkout <hash-du-commit> vous téléporte littéralement dans le passé de votre projet. La commande git revert crée un nouveau commit qui annule les effets d'un commit précédent de façon propre.

Grâce aux branches, chaque développeur peut travailler sur sa propre fonctionnalité de façon complètement isolée. Alice développe le module de paiement pendant que Bob corrige des bugs — sans aucune interférence.

Lorsque les deux ont terminé, Git fusionne intelligemment les modifications. En cas de conflit sur la même ligne d'un fichier, Git vous demande de décider quelle version conserver.

Chaque commit enregistre l'auteur, la date et un message descriptif. La commande git blame vous permet de voir, ligne par ligne, qui a écrit chaque portion de code et dans quel commit.

C'est un outil indispensable pour comprendre le pourquoi derrière chaque décision de code, surtout sur des projets de longue durée avec de multiples contributeurs.

Vous pouvez créer une branche pour tester une idée radicale sans aucun risque pour le code de production. Si l'expérience échoue, supprimez simplement la branche. Si elle réussit, fusionnez-la.

Cette liberté d'expérimentation est l'une des raisons pour lesquelles Git favorise l'innovation : le coût de l'échec est quasiment nul.

Git est au cœur de toutes les pratiques DevOps modernes. Les pipelines CI/CD (Intégration Continue / Déploiement Continu) sont déclenchés par des événements Git : push, pull request, tag de release…

GitHub Actions, GitLab CI, CircleCI, Jenkins — tous ces outils s'appuient directement sur les événements Git pour automatiser les tests, le build et le déploiement de vos applications.
